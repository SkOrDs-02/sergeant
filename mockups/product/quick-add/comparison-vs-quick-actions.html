quick-add vs QuickActionsMenu — два інструменти, не конкуренти

Один створює новий запис із Hub-рівня (FAB → нижній лист). Другий діє на існуючий елемент списку (long-press → радіальне меню). Це різні рівні UI — обидва можуть жити, треба тільки чітко розмежувати скоупи.

КонтекстCouncil audit · Q2 prototype Дата2026-05-18 Файлиmockups/product/quick-add/index.html · apps/web/src/shared/components/ui/QuickActionsMenu.tsx
Verdict

Це не «один замість іншого» — це різні рівні

quick-add = «створити новий запис у будь-який модуль» (Hub entry point). QuickActionsMenu = «зробити дію над існуючим item» (per-row context). Якщо ship обидва — потрібен документальний boundary. Якщо ship лише quick-add — QuickActionsMenu deprecate. Найдешевший шлях: спочатку quick-add (Phase 4), а QuickActionsMenu залишити orphan-у-shared/ до моменту, коли live-product покаже потребу у per-item long-press.

створити quick-add · FAB → лист

Юзер хоче додати новий запис — не важливо в який модуль. Тап по FAB на Hub-екрані → відкривається нижній лист з 4-modulе picker + 4 mode tiles. 2 тапи замість переходу в модуль.

Обидва на одному екрані — як уявити

FAB унизу-справа — це quick-add вхід. Long-press на будь-якому рядку списку — це QuickActionsMenu. Між ними немає конфлікту: різні жести, різні цілі, різний рівень UI.

  • tap FAB → нижній лист «куди + як записати» → створення
  • long-press item → радіальне меню над item'ом → дія над існуючим

Ризик розрулювати — щоб юзер не плутав. Tutorial spotlight на FAB при першому Hub-візиті + onboarding-hint про long-press при першому open списку. Tech-debt не виникне, якщо документуємо це у backlog.md при ship'і Phase 4 FAB.
