# Спека: персистентність даних незалогіненого користувача

> **Last touched:** 2026-07-22 by @Skords-01. **Next review:** 2026-08-22.
> **Status:** Active — рішення Р1–Р3 ухвалені 2026-07-22, спека готова до імплементації.

## Проблема

**Дані незалогіненого користувача не переживають перезавантаження сторінки.**
Створена звичка / витрата існує лише в теплому кеші поточної сесії; після
`reload` її немає.

Це ламає обіцянку, яку продукт дає явно: `/sign-in` пропонує «Поки що
пропустити», а `/welcome` веде анонімного візитера прямо в роботу. Перше, що
така людина створює, зникає при наступному відкритті застосунку — **тихо, без
помилки**.

## Доказ (виміряно 2026-07-22, dev + живий API, анонімна сесія)

Шлях: `/welcome` → «Почати» → СТАРТ → «Рутина: Створи першу звичку» → пресет.

| Крок                                                | Результат                |
| --------------------------------------------------- | ------------------------ |
| Одразу після створення, in-app перехід у `/routine` | **`0/1`** — звичка є     |
| Той самий екран після `reload`                      | **`0/0`** — звички немає |

Ручний шлях (`/routine` → «Додати звичку» → поле «Назва» → зберегти) дає
**той самий результат**: `0/1` до перезавантаження, `0/0` після.

Тобто дефект **не в пресет-шиті** — він у шарі персистентності.

## Механізм

1. [`useRoutineDualWriteBoot.ts:36`](../../../../apps/web/src/modules/routine/hooks/useRoutineDualWriteBoot.ts) —
   `if (!userId) return;`. Для анонімного користувача контекст dual-write
   **не реєструється**.
2. [`sqliteWriter/index.ts:199`](../../../../apps/web/src/modules/routine/lib/sqliteWriter/index.ts) —
   `triggerRoutineDualWrite`: `if (!registeredContext) return;`. Без контексту
   запис у SQLite не відбувається взагалі.
3. [`routineStorage.ts:139`](../../../../apps/web/src/modules/routine/lib/routineStorage.ts) —
   `saveRoutineState` встигає оновити **теплий кеш** (`setCachedSqliteRoutineState`),
   тому UI одразу показує запис. На boot кеш перебудовується з SQLite, де
   запису немає.
4. Старий LS-шлях (`hub_routine_v1`) затомбстоунено на Stage-8 — у
   [`presetApply.ts:17`](../../../../apps/web/src/core/onboarding/presetApply.ts)
   прямо сказано, що `applyRoutinePreset` більше туди не пише.

Тобто після міграції на SQLite анонімний користувач лишився **без жодного
довговічного шляху запису**.

### Межі

Гейт `if (!userId) return;` ідентичний у всіх чотирьох модулях:
`useRoutineDualWriteBoot`, `useFizrukDualWriteBoot`,
`useNutritionDualWriteBoot`, `useFinykDualWriteBoot`.

**Виміряно наживо тільки routine.** Для решти висновок зроблено з коду —
перед фіксом варто підтвердити хоча б одним модулем іще.

Читальний шлях має half-fallback: `useSqliteReadBoot` та аналоги підставляють
`DEMO_LOCAL_USER_ID` **лише в demo-режимі** (`user?.id ?? (isDemoActive() ?
DEMO_LOCAL_USER_ID : null)`). У write-шляху fallback-у немає взагалі — навіть
для demo. Demo виглядає робочим лише тому, що його дані сідяться в LS і
підтягуються `residualImport` на boot.

## Рішення — УХВАЛЕНО 2026-07-22

> Ратифіковано дизайн-аудитом (`cycle-3-verdicts-2026-07-22.md` §1).
> Формулювання питань лишені нижче як контекст.

**Р1 → сталий синтетичний id `local-anon`**, за зразком `DEMO_LOCAL_USER_ID`,
однаковий у read- і write-boot усіх чотирьох модулів. Окремо: **write-шлях
demo теж отримує fallback на `DEMO_LOCAL_USER_ID`** — те, що demo «виглядає
робочим» лише через LS-residual, це прихована бомба того самого класу.

**Р2 → (а) міграція при першому логіні.** Варіант (б) перекладає наш борг на
користувача, (в) вбиває воронку 3 кліки / 0 полів — єдину сильну ланку
перших 60 секунд. Правило: при першому успішному логіні рядки `local-anon`
переписуються на реальний `userId`, після успішного перенесення
`local-anon`-рядки видаляються. Міграція **ідемпотентна** (повторний логін —
no-op). Тригер — перший авторизований boot, не submit форми.

**Р3 → merge за LWW `clientTs`, ніколи не відмова.** Той самий контракт, що
dual-write. Діалог «вибери набір даних» заборонений: людина, яка попрацювала
анонімно і залогінилась, не мусить ухвалювати рішення про синхронізацію.

**Похідне правило:** СТАРТ-блок позначається витраченим лише після
підтвердженого **durable** write (SQLite під `local-anon` або `userId`); на
фейлі блок лишається + видима помилка наявною мовою станів.

### Контекст: як питання формулювались

Це не механічний фікс — тут вибирається семантика ідентичності.

**Р1. Під якою ідентичністю пише анонім?**
Найдешевший варіант у наявному патерні — сталий синтетичний id за зразком
`DEMO_LOCAL_USER_ID` (напр. `local-anon`), однаковий для read- і write-boot
у всіх модулях.

**Р2. Що стається з цими даними при вході в акаунт?** — головне питання.
Без відповіді фікс Р1 просто переносить втрату на пізніше: людина попрацює
анонімно, залогіниться і побачить порожньо.

- (а) міграція: при першому успішному логіні перескопувати рядки з `local-anon`
  на реальний `userId`;
- (б) без міграції, але з чесним попередженням в UI до логіну;
- (в) заборонити анонімний запис узагалі — прибрати «Поки що пропустити».
  Найгірший для воронки: зараз вона 3 кліки / 0 полів саме завдяки анонімності.

**Р3. Конфлікти.** Якщо на акаунті вже є дані, а локально теж — merge чи
відмова? Тут діють ті самі LWW-правила, що й у dual-write (`clientTs`), але
рішення треба зафіксувати явно.

## Definition of done

- Р1–Р3 ухвалені й записані сюди.
- Анонімний запис переживає reload у всіх чотирьох модулях (тест на кожен).
- E2E-регресія: створити запис анонімно → reload → запис на місці.
- Якщо обрано міграцію (Р2а) — окремий тест «анонімно створив → залогінився →
  дані на місці».
- Прогін під логіном і на чистому rate-limit (у вимірі 2026-07-22 консоль
  віддавала 401×2/429/503 — це фон анонімної сесії, не причина дефекту, але
  межі варто підтвердити).
