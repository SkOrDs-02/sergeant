1. **Стрік:** щоденні послідовні дні в коді ↔ «має бути тижневим/гнучким».

3. **Травма:** recovery радить без поняття «не можна» ↔ «додати в модель».
4. **1RM:** LoadCalculator від пікової ↔ «старіння + протокол повернення».
5. **План:** власник fizruk ↔ код у routine (FizrukDayPlanSheet).
6. **JTBD:** «будь-яка фізактивність» ↔ суто силова модель даних.

## Додаток А: транскрипт інтерв'ю (2026-07-23, дослівно)

**Раунд 1**

- (а) Хто користувач і JTBD? → **«Будь-яка фізактивність — ціль»** (силові —
  перша реалізація, не межа).
- (б) Успіх? → **«По собі суб'єктивно / не визначав»**.
- (в) Межі? → **«Не людський тренер/методичка»** (єдина вибрана).
- Власник планування тренувань? → **«Fizruk — план його, код дрейфнув»**
  (у routine; історичний артефакт, колись повернути).

**Раунд 2**

- Розрив «будь-яка активність» vs силова модель? → **«Силові спочатку,
  інші типи — потім»** (модель доведеться розширювати — напрям).
- Recovery-движок? → **«Ключова фіча — підказує ЩО тренувати»**
  (диференціатор, розвивати).
- Стрік (щоденний карає за відпочинок)? → **«Стрік має бути
  тижневим/гнучким»** (щоденна логіка — помилка для фітнесу, міняти).
- Health-інтеграції та соціальне? → **«Обидва — майбутні напрями»**
  (health: пульс/сон для recovery; соціальне — теж не «ніколи»).

**Раунд 3 (за blindspot-пасом)**

- Травма/біль (recovery порадить «груди» з розтягнутим плечем)? →
  **«Додати "не можна" в модель»** (позначка болю через wellbeing/atlas →
  recovery виключає м'язи; медичних порад не даємо — лише не радимо
  травмоване).
- Конфлікт стимулів (digest хвалить об'єм ↔ recovery каже відпочинь)? →
  **«Recovery головний — хвалити розумне, не більше»** (digest/coach
  узгоджують похвалу з recovery-станом; хвалити відпочинок у день
  відпочинку; тоннаж — внутрішня цифра, не похвала).
- Регрес після перерви (пікова 1RM у LoadCalculator)? → **«Старіння 1RM +
  протокол повернення»** (після N тижнів 1RM застаріла; м'який режим без
  порівняння з піком; повернення — окремий продуктовий стан).
- Програми: вести чи каталог? → **«Вести з прогресією — це не "тренер"»**
  (механічна прогресія — частина цінності; межа «не тренер» =
  не індивідуальна методологія/медицина).

**Раунд 4 (за blindspot-пасом)**

- Одиниця активності при розширенні? → **«Не рівні — типи матимуть вагу»**
  («повноцінне тренування» ≠ «легка активність» для стріку/digest/активації;
  семантику закласти до розширення).
- Власник ваги тіла (fizruk ↔ nutrition)? → **«Fizruk володіє тілом,
  nutrition читає»** (одне джерело істини).
- Критерій помилки recovery? → **«Поки вірю на слово — критерію немає»**
  (чесна невизначеність; recovery — завжди порада, не гейт).
- (е) Невпевненість? → **«Чи правильна recovery-евристика» + «Чи цінний
  Atlas (теплова мапа)»**.

## Додаток Г: blindspot-пас по банку питань (2026-07-23, Fable, стисло)

Питання: «яке продуктове питання банк не ставить?» (класи (є)/(ж) і
«що відбувається з людиною» виключені як уже стандартні). 7 знахідок,
усі відпрацьовані (раунди 3-4):

1. **Фізична шкода** — травма/біль у моделі поради. → «додати "не можна"».
2. **Конфлікт стимулів** — digest хвалить об'єм ↔ recovery каже менше.
   → «recovery головний».
3. **Регрес/повернення** — застаріла 1RM як активний вхід у пораду ваги.
   → «старіння + протокол повернення».
4. **Епістемологія recovery** — критерій коректності радника. → «критерію
   немає» (чесна невизначеність).
5. **«Не методичка» vs каталог програм** — межа ведення. → «вести з
   прогресією ок».
6. **Одиниця активності** — еквівалентність типів при розширенні. → «типи
   матимуть вагу».
7. **Власник тіла** — вага між fizruk і nutrition. → «fizruk володіє,
   nutrition читає».

Мета-спостереження: fizruk — єдиний модуль, де порада продукту може
завдати ФІЗИЧНОЇ шкоди (травма, вага на штанзі) — клас «безпека поради»
додати до шаблону для модулів-порадників.
