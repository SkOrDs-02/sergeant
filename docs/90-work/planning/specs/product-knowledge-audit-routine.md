# Продуктовий аудит знань: routine

> **Last touched:** 2026-07-24 by @Skords-01 (інтерв'ю: founder × Claude Fable 5; blindspot-пас по банку питань виконано до фіналізації).
> **Status:** Draft

Самодостатня спека для окремої сесії-виконавця — п'ятий (останній модульний)
прогін конвеєра (finyk, hub-coach, nutrition — виконані, канони в
`docs/01-product/model/`; fizruk — спека в роботі). Предмет — модуль
**routine** (звички). Founder-колонка ВЖЕ зафіксована в Додатку А (4 раунди)
— виконавця вона не стосується до етапу зведення.

Контекст входу: founder найменше розуміє цей модуль і відчуває його «пустим
та голим» — при тому, що **користується ним щодня**. Діагноз цього відчуття
(глибина/поліш, Додаток А раунди 2-3) — частина предмета аудиту.

## 1. Мета й non-goals

**Аудит доводить (або спростовує):** founder, документація і код розуміють
routine однаково. Розбіжність між колонками — знахідка з наслідком (§6).

**Non-goals:**

- НЕ рефакторинг і НЕ виправлення знахідок — лише фіксація + рішення.
  Кодові зміни виконавцю ЗАБОРОНЕНІ (критерій §8.7).
- НЕ аудит інших модулів.
- НЕ security/perf.
- НЕ зміна GTM/прайсингу.

**Артефакти:**

1. `docs/01-product/model/routine.md` — канон (§2).
2. Diff-звіт — `docs/90-work/audits/product-knowledge-routine.md`.

## 6. Формат diff-звіту

Ідентичний попереднім: «питання × FOUNDER × DOCS × CODE × вердикт ×
наслідок» + рішення на кожну розбіжність.

Наперед відомі напруги (обов'язкові рядки):

1. **Стрік:** жорсткий без заморозок у коді ↔ «гнучкий + планована пауза
   наперед + пропуск із причиною».
2. **Чистота модуля:** «чистий трекер звичок» ↔ FizrukDayPlanSheet усередині
   - pushup-шов (фізактивність належить fizruk).
3. **Streak-at-risk push** обіцяний у GTM ↔ у коді немає; роль нагадувань
   не вирішена.
4. **Тір-гейти нагадувань** (Free 2 / Plus 5 / Premium ∞) у доках ↔ гейтів
   у коді немає.
5. **Mobile:** маркетинг описує стріки/heatmap без platform-скоупу ↔
   mobile — заглушка «Скоро буде портовано».
6. **Once/todo:** межа «не todo-менеджер» ↔ once-розклад +
   useTodoEveningInsight у коді; перевірити, чи once впливає на
   стрік/heatmap/rate (founder: не має).

## Додаток А: транскрипт інтерв'ю (2026-07-23/24, дослівно)

**Раунд 1**

- (е-чесне) Чи користуєшся сам щодня? → **«Так, користуюсь щодня»**.
- (а) Продуктова роль? → **«Чистий трекер звичок»** (FizrukDayPlanSheet —
  випадковий гість, має переїхати).
- Стрік-філософія? → **«Гнучкий всюди — додати заморозки»** (жорсткий
  демотивує; єдина м'яка філософія продукту).
- (б) Успіх? → **«Щоденний чекін у модуль»**.

**Раунд 2**

- Що саме «пусте»? → **«Відмітив і вийшов — нема глибини» + «UI
  голий/недопрацьований»** (обидві причини).
- (в) Межі? → **«Не todo/таск-менеджер» + «Не коучинг формування звичок»**.
  Гейміфікація і календар-синк свідомо не вибрані — статус відкритий.
- Роль нагадувань (streak-at-risk, тір-гейти)? → **«Не вирішив»**.
- Mobile-заглушка? → **«Чинна політика — PWA закриває телефон»**.

**Раунд 3 (за blindspot-пасом)**

- Семантика заморозки? → **«Планована пауза наперед»** (відпустка/хвороба —
  дні паузи не рахуються взагалі).
- Причина пропуску? → **«Так — пропуск з причиною»** (тристанова модель:
  зробив / не зміг з причиною / не зробив; heatmap і digest розрізняють).
- Глибина — це що? → **всі чотири: «Контекст/нотатка до відмітки» +
  «Інсайти/кореляції в stats» + «Емоційна винагорода моменту» +
  «UI-поліш загалом»**.
- Дім ритуалу? → **«Обидва рівноцінні»** (routine-календар і hub-картка —
  першокласні й синхронні).

**Раунд 4 (за blindspot-пасом)**

- Once/todo? → **«Once лишається — легкий виняток»** (у межах дня, без
  дедлайнів/проєктів; стрік/heatmap для once не рахуються).
- Pushup-шов? → **«Ні — фізактивність належить fizruk»** (віджимання-звичка
  має стати fizruk-активністю легкого типу, routine лише нагадує; шов
  розібрати).
- WIP-ліміт звичок? → **«Без позиції — скільки хочеш»**.
- (е) Невпевненість? → **«Роль нагадувань і їх монетизація» + «Чи працюють
  стріки як мотивація взагалі» + «Чи потрібні категорії/теги»**.

## Додаток Г: blindspot-пас по банку питань (2026-07-23, Fable, стисло)

Питання: «яке продуктове питання банк не ставить?» (стандартні класи
(є)/(ж)/«людина»/«безпека поради» виключені). 7 знахідок, усі
відпрацьовані (раунди 3-4):

1. **Семантика і бюджет заморозок** — «гнучкий» без специфікації. →
   планована пауза наперед.
2. **Розрізнення «не зміг»/«не зробив»** — бінарний completion бреше. →
   пропуск із причиною.
3. **Що таке «глибина»** — скарга без специфікації. → всі 4 складові.
4. **Дім ритуалу** — routine-таб чи hub-картка. → обидва рівноцінні.
5. **Once vs «не todo»** — троянський todo в enum. → once-виняток без
   стріку.
6. **Pushup-шов** — data-залежність, яку переїзд Sheet не покриває. →
   розібрати, фізактивність fizruk.
7. **Ємність дня (WIP-ліміт)** — властивість набору звичок. → без позиції.

Мета-спостереження (нове для конвеєра): routine має клас дір **«рішення
прийняті, але порожні всередині»** — попередні модулі мали невирішені
питання, тут рішення («гнучкий стрік», «нема глибини») існували без
доменної моделі. Знахідки 1-3 — один взаємозалежний кластер; при
тиражуванні на інші продукти перевіряти прийняті рішення на
специфікованість, не лише наявність.
