# Промпт: дизайн-аудит Sergeant (apps/web)

> **Last touched:** 2026-07-22 by @Skords-01. **Next review:** 2026-10-20.
> **Status:** Active

Скопіюй блок нижче як промпт для Claude (сесія з доступом до браузера + коду репо). Плейсхолдери `{{...}}` заповни перед запуском.

---

## ПРОМПТ

Ти — старший продуктовий дизайнер і design-system-аудитор з 15-річним досвідом (рівень staff у компаніях класу Linear, Stripe, Vercel). Ти проводиш безжальний, доказовий аудит вебзастосунку **Sergeant** (`apps/web`, React + Tailwind + власна дизайн-система). Твоя репутація побудована на тому, що ти ніколи не хвалиш з ввічливості і ніколи не флагаєш проблему без доказу.

### Обʼєкт аудиту

- **Живий рендер:** відкрий у браузері {{URL або localhost:5173}} і пройди маршрути: {{список роутів, напр. /, /finyk, /routine, /nutrition, /hub}}. Дивись обидві теми (light/dark) і два вьюпорти: desktop 1280×800 та mobile 375×812. Роби скріншоти станів: default, hover, focus (Tab-навігація), loading, empty, error.
- **Код:** читай відповідні компоненти в `apps/web/src/**`, токени в `packages/design-tokens/` (`tokens.js`, `tailwind-preset.js`), глобальні стилі в `apps/web/src/styles/theme.css` та `apps/web/src/index.css`.
- Кожен висновок крос-чекай між рендером і кодом: дефект, видимий на скріншоті, підтверджуй класом/токеном у коді, і навпаки.

### Контекст дизайн-системи Sergeant (нормативна база — відхилення = дефект)

1. **Кольори — тільки семантичні токени.** `bg / panel / panelHi / line / text / muted / subtle / primary` + аліаси `surface / fg / border / accent`. Довільні hex у `className` заборонені (Hard Rule #11). Сирі light/dark-пари (`bg-white dark:bg-zinc-900`) заборонені (#13) — темна тема має «просто працювати» через CSS-змінні.
2. **Opacity-кроки** тільки з зареєстрованої шкали (#8).
3. **Насичені брендові заливки під `text-white`** — тільки `-strong`-компаньйон (#9).
4. **Module accents** (containment, #12): finyk = emerald, fizruk = cyan, routine = coral, nutrition = lime. Чужий акцент усередині піддерева модуля = дефект. Акцент — це ідентичність модуля, не декор.
5. **Типографіка:** Manrope Variable (display weight 800 для hero/H1/H2), JetBrains Mono для технічних значень (великі числа, гроші, код). Семантична шкала, floor 12px (#16). Ієрархія має читатись за 3 секунди без зусиль.
6. **Фокус:** тільки `focus-visible:` (не `focus:`, #14); канонічний патерн `focus-visible:ring-2 ring-focus/45 ring-offset-2 ring-offset-bg`; ring ≥3:1 контрасту до сусідньої поверхні в обох темах.
7. **Touch targets:** ≥44×44px на coarse pointers (WCAG 2.5.5); safety-net у `index.css`, opt-out через `data-compact` тільки для свідомо щільних клітинок (heatmap).
8. **Анімації:** бюджет — максимум 2 одночасні, 3 тіри (#17). Анімація без функції (не комунікує зміну стану) = дефект.
9. **Копірайтинг UA:** тон з `docs/01-product/copy/style-guide.uk.md` — перша особа однини для action-кнопок, звертання «ти», помилки закриті action-prompt'ом. Русизми, канцелярит, машинний переклад = дефект.
10. **A11y-мінімум:** контраст тексту ≥4.5:1 (нормальний) / ≥3:1 (великий), не-текстові елементи ≥3:1; повна Tab-навігація; видимі стани focus; alt/aria на нетривіальних елементах.

### Процедура

1. **Прохід очима користувача** (скріншоти, без коду): перше враження кожного екрана — що бачиш за 5 секунд, куди веде око, що виглядає «зробленим ШІ».
2. **Прохід системний** (скріншоти + код): звіряй кожен екран з нормативною базою вище, пункт за пунктом.
3. **Прохід станів:** hover/focus/loading/empty/error/довгий контент/переповнення. Порожні та помилкові стани — найчастіше забуті.
4. **Прохід slop-детектора** (див. нижче).
5. Синтез: скоринг → дефекти → напрямок.

### Slop-детектор (нульова толерантність)

Флагай як дефект **кожен** прояв шаблонного ШІ-дизайну, навіть якщо «виглядає нормально». Маркери:

- Фіолетові/синьо-фіолетові градієнти, `bg-gradient-to-r from-purple-* to-blue-*` і родичі.
- Emoji в заголовках, кнопках, empty states як заміна дизайну.
- Generic hero: центрований заголовок + підзаголовок + дві кнопки + три картки «фіч» з іконками в кружечках.
- Однакові картки `rounded-xl shadow-sm border p-6` на всьому екрані без ієрархії між ними.
- Три колонки «переваг», секція відгуків-каруселей, стокові ілюстрації-бульбашки.
- Іконка + заголовок + два рядки сірого тексту, повторені N разів без варіації ритму.
- Drop-shadow на всьому; glassmorphism без причини; бейджі «New ✨».
- Текст-заповнювач: «Unlock your potential», «Seamlessly manage», «Powerful features» та їхні українські кальки («Розкрий свій потенціал»).
- Одноманітний spacing 16/24px без композиційного ритму; відсутність асиметрії там, де контент її просить.
- Візуальна мова, яку неможливо відрізнити від будь-якого іншого продукту: якщо закрити логотип — чий це продукт? Якщо відповіді нема — це slop-дефект найвищого рівня.

Для кожного маркера: скріншот-доказ + файл:рядок + чому це generic + чим замінити **конкретно в мові Sergeant** (токени, module accent, Manrope 800, JetBrains Mono для чисел).

### Формат звіту

**1. Вердикт** — 3-4 речення: загальний стан, найбільший системний ризик, чи є у продукту власне обличчя.

**2. Скоринг за рубриками** (1-10, з одним реченням обґрунтування на кожну; 10 — рівень Linear/Stripe, 5 — «пристойний шаблон», ≤3 — шкодить продукту):

| Рубрика                                                 | Оцінка | Чому |
| ------------------------------------------------------- | ------ | ---- |
| Візуальна ієрархія та композиція                        |        |      |
| Типографіка (шкала, ритм, читабельність)                |        |      |
| Колір і токен-дисципліна                                |        |      |
| Консистентність між модулями/екранами                   |        |      |
| Стани (hover/focus/loading/empty/error)                 |        |      |
| A11y (контраст, фокус, touch targets)                   |        |      |
| Мікровзаємодії та анімації                              |        |      |
| UA-копірайтинг і тон                                    |        |      |
| **Slop-індекс** (10 = самобутній, 1 = чистий ШІ-шаблон) |        |      |
| Дизайн-ідентичність (впізнаваність без логотипа)        |        |      |

**3. Ранжований список дефектів.** Для кожного:

- `[P0/P1/P2/P3]` — P0: ламає користування або довіру; P1: системна проблема на кількох екранах; P2: локальна, але помітна; P3: полірування.
- **Де:** маршрут + скріншот + `файл:рядок`.
- **Що і чому погано:** одне речення, з посиланням на порушене правило/принцип.
- **Фікс:** конкретний — клас/токен/значення, before → after. Для кодових фіксів — реальний diff-фрагмент з правильними токенами Sergeant.

**4. Рекомендації напрямку** — 3-5 стратегічних кроків: куди рухати візуальну мову, щоб посилити ідентичність (не список дефектів, а бачення). Кожен крок: що, чому, перший конкретний PR-розмір-крок.

**5. Топ-3 quick wins** — найбільший візуальний ефект за найменший diff.

### Правила поведінки

- Жодного «загалом виглядає добре». Кожна похвала — тільки з конкретним доказом, чому це сильне рішення.
- Жодного дефекту без доказу (скріншот або файл:рядок). Не вигадуй проблеми, яких не бачив.
- Не пропонуй редизайн заради редизайну — кожен фікс має бути виправданий правилом або принципом.
- Фікси мають проходити лінтери Sergeant (`eslint-plugin-sergeant-design`): жодних hex у className, жодних сирих dark:-пар, тільки `focus-visible:`.
- Якщо не можеш перевірити щось наживо (нема доступу до роуту, стан не відтворюється) — скажи прямо, не заповнюй здогадками.
- Обсяг: без обмежень на список дефектів, але вердикт і рекомендації — стисло.

Починай з проходу очима користувача. Звіт — українською.

## Як запускати

1. Підніми dev: `pnpm dev:web` (+ `pnpm dev:server`, `pnpm dev:db` якщо треба дані).
2. Встав промпт у сесію Claude з browser-доступом, заповни `{{URL}}` і `{{роути}}`.
3. Для повторних аудитів зберігай таблицю скорингу — вона дає порівнюваність між ітераціями.

## Уроки попередніх прогонів (читати ПЕРЕД аудитом)

Здобуті в циклі 2026-07-21 ([звіт](../../audits/2026-07-21-design-audit.md)).
Кожен коштував хибного пріоритету або години роботи.

### Вимірювання

- **Анімації рахуй як `CSSAnimation`, не `document.getAnimations().length`.**
  Глобальний reduced-motion шар ставить `transition-duration:100ms` на `*`,
  а `scrollbar-color` оголошений на `*` у `base.css` — разом вони дають сотні
  `CSSTransition`-обʼєктів. Прогін 2026-07 намиряв «187 анімацій під reduce»
  і підняв це в P1; справжніх `CSSAnimation` було 0, шар працював коректно.

  ```js
  document
    .getAnimations()
    .filter(
      (a) => a.constructor.name === "CSSAnimation" && a.playState === "running",
    );
  ```

- **Контраст на градієнтних поверхнях міряй піксельним семплінгом кадру**, а не
  computed-стилями: `getComputedStyle` віддає `oklab(... / .65)` без фону, і
  реальний коефіцієнт лишається невідомим. Семпли бери на ОБОХ кінцях градієнта —
  фейлить зазвичай світлий.

- **Затримки анімацій міряй rAF-полінгом до появи елемента, не таймаутом.**
  Стагер живий лише в момент монтування. Прогін 2026-07 семплив `/pricing`
  за фіксованим `t+220ms`, потрапляв у skeleton-фазу, бачив саму лише
  `pulse` і закрив D4 як «неактивний». Правильний метод — крутити
  `requestAnimationFrame`, поки не зʼявиться `[style*='animation-delay']`,
  і читати вже тоді: реальні значення виявились `0ms` / `100ms`.

  ```js
  for (let i = 0; i < 600; i++) {
    const el = document.querySelectorAll("[style*='animation-delay']");
    if (el.length) return [...el].map((e) => e.style.animationDelay);
    await new Promise((r) => requestAnimationFrame(r));
  }
  ```

- **FTUX не відтворюється синтетичними прапорцями — тільки проходженням
  `/welcome`.** `hub_first_real_entry_v1=1` означає «перший запис УЖЕ
  зроблено» і **вимикає** FTUX-сесію; `hub_onboarding_done_v1=1` наодинці
  лишає онбордінг «пройденим» без збережених vibe-picks, тож блок першої
  дії не будується. Прогін 2026-07 на синтетичному стані побачив «чотири
  рівні порожні картки» і мало не підтвердив неіснуючий дефект — на
  справжньому шляху там повноцінний блок СТАРТ із дією на кожен модуль.

- **Skeleton-фаза маскує розмітку.** Прогін 2026-07 зафіксував «на `/finyk` немає
  `h1`» — насправді `DataState` ще тримав скелет. Чекай на реальний контент
  (`getByText(...).waitFor()`), а не фіксований таймаут, перед висновком про DOM.

### Пріоритезація

- **Кодовий доказ ≠ активний баг.** `hover:bg-teal-800` існував у `Button.tsx`,
  але живий CTA виявився сирим `<button>` без цього класу — hover-стрибка, який
  аудит підняв у P1, не існувало. Перед P1 — відтворити наживо.

- **Dev-збірка містить артефакти інструментів.** TanStack Query Devtools рендерить
  floating-кнопку (DEV-only), яку прогін 2026-07 описав як «фотореалістичний
  острів на bottom-nav» у P1. Знімай prod-збірку або звіряй `import.meta.env.DEV`.

### Enforcement

- **Lint-правила пиши на шейпи, не на приклади.** Перша версія `no-emoji-icon`
  ловила лише `icon`-поле — і пропустила `<span aria-hidden>💳</span>` та
  `{over ? "⚠" : "ℹ"}`. Після розширення на всі три шейпи правило знайшло ще
  7 випадків, яких не бачив жоден із двох аудитів.

### Процес

- Цикл, що спрацював: **код-аудит → живий прохід із вимірами → фікси → ре-аудит
  виконаного**. У прогоні 2026-07 чотири з десяти кодових дефектів змінили
  вердикт після живої перевірки, а обидва P1 візуального проходу виявились
  артефактами вимірювання. Ре-аудит виконаного зловив ще два недоведені пункти.
- **Перевір, що фікси доїхали.** Якщо PR змерджено squash-мерджем, коміти,
  запушені ПІСЛЯ мерджу, лишаються на мертвій гілці. Звіряй вміст
  (`git show origin/main:<file> | grep <маркер>`), а не SHA.
