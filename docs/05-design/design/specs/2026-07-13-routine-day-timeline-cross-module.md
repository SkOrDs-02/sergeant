# Routine — timeline дня та кросмодульний календар

> **Last touched:** 2026-07-19 by @claude. **Next review:** 2026-10-17.
> **Status:** Active

> **Статус:** Draft — product interview complete, implementation not started.
> **Дата:** 2026-07-13
> **Джерело:** інтервʼю з фаундером у поточній робочій сесії.
> **Канонічна роль Routine:** трекер звичок і керований календар дня, який показує важливі події з Finyk та Fizruk.

## 1. Проблема

Зараз Routine сприймається як мінімальний календар/трекер без очевидної відповіді на три питання:

1. Що тут можна робити?
2. Для чого Routine потрібна в Sergeant?
3. Як вона повʼязана з іншими модулями?

Поточний код уже має календарну основу, статистику, власні звички, Finyk-підписки та Fizruk-план. Але ці можливості не складаються в одну зрозумілу модель продукту: користувач бачить окремі блоки й фільтри, а не порядок власного дня.

## 2. Висновки інтервʼю

Зафіксовані рішення:

| Питання                   | Рішення                                                                      |
| ------------------------- | ---------------------------------------------------------------------------- |
| Основна роль              | Трекер звичок + зведений календар модульних подій                            |
| Перший екран              | Хронологія дня, а не ізольований список звичок                               |
| Події першої версії       | Фінансові події, тренування, власні звички                                   |
| Джерело подій             | Автоматичне підтягання з модулів                                             |
| Контроль джерел           | Користувач обирає підключені модулі на старті та може змінити вибір пізніше  |
| Події без часу            | Окремий блок «Без часу»                                                      |
| Підтвердження чужих подій | Відбувається у власному модулі, не в Routine                                 |
| Підтвердження звички      | Швидка дія в timeline або детальний екран Routine                            |
| Довільний планер          | Не входить у першу версію; Routine поки не створює довільні календарні події |
| Перший запуск             | Спочатку налаштування джерел, потім timeline                                 |

## 3. Продуктова модель

> **Routine показує, що формує день користувача, допомагає не загубити власні звички та веде до правильного модуля для виконання дії.**

Routine не стає універсальним task manager і не копіює чужі доменні редактори.

### Власні дані Routine

- звички;
- розклад повторення;
- час виконання, якщо він заданий;
- стан виконання, пропуску або паузи;
- нотатка й історія виконання;
- streak та агрегована статистика.

### Агреговані дані

- Finyk: фінансові події, першою чергою регулярні платежі та підписки;
- Fizruk: заплановані тренування та повʼязані календарні події;
- майбутні джерела можуть додаватися через той самий контракт, але не входять до MVP.

Для агрегованої події джерельний модуль залишається власником стану. Routine зберігає проєкцію для показу та deep-link назад у джерело.

## 4. Користувацькі сценарії

### 4.1. Перший запуск

1. Користувач відкриває Routine.
2. Бачить коротке пояснення: «Routine збирає звички та важливі події твого дня в одному календарі».
3. Обирає джерела, які хоче бачити: Finyk, Fizruk.
4. Бачить короткий preview: які типи подій буде підключено.
5. Потрапляє в timeline дня.
6. Вибір можна змінити в налаштуваннях Routine.

Якщо користувач не підключив жодного джерела, екран не повинен виглядати порожнім: показуємо власні звички, пояснення ролі модулів і CTA для підключення.

### 4.2. Перегляд дня

Timeline сортує події за Kyiv day key і показує:

1. події з конкретним часом — від ранку до вечора;
2. окремий блок «Без часу»;
3. завершені події — згорнуто внизу, щоб не створювати шум.

Кожен рядок має джерело, назву, час або label «Без часу», короткий стан і дію.

### 4.3. Робота зі звичкою

- натискання на швидку дію відмічає власну звичку виконаною;
- відкриття рядка або кнопки деталей показує habit detail;
- detail дозволяє переглянути історію, нотатку, streak, пропустити/перенести або змінити розклад;
- дія оновлює лише дані Routine.

### 4.4. Робота з подією Finyk/Fizruk

- рядок у Routine є read-only проєкцією;
- натискання веде у відповідний модуль із вибраною подією/датою;
- підтвердження, редагування або завершення відбувається у Finyk чи Fizruk;
- після зміни джерельні дані повторно відображаються в Routine;
- Routine не створює дубль і не показує власний окремий статус, який може розійтися з джерелом.

## 5. Інформаційна архітектура

### Основний екран

- заголовок Routine з коротким поясненням;
- date navigator: сьогодні, попередній/наступний день, повернення до сьогодні;
- day progress: виконані власні звички / заплановані власні звички;
- timeline подій;
- блок «Без часу»;
- згорнутий блок завершених подій;
- нижня навігація до календаря та статистики.

### Фільтри

- швидкі фільтри джерел: Routine, Finyk, Fizruk;
- фільтри зберігають лише локальний UI-стан;
- постійне підключення джерел змінюється в налаштуваннях;
- вимкнення джерела прибирає його проєкцію з Routine, але не змінює дані модуля.

### Календарні режими

MVP зберігає наявні режими day/week/month, але day/timeline є default experience. Week і month потрібні для огляду повторюваності, планів і streak, а не як головний landing state.

## 6. Керування підключеннями

Налаштування Routine повинні мати окремі контролі:

- **Показувати фінансові події** — Finyk;
- **Показувати тренування** — Fizruk;
- у майбутньому — окремі підтипи подій, якщо один модуль має надто багато шуму.

Поведінка:

- перший запуск: жодне джерело не вважається підключеним без вибору користувача;
- повторний запуск: використовуємо збережені налаштування;
- вимкнення: не видаляє джерельні дані й не змінює Finyk/Fizruk;
- відсутність даних або недоступне джерело: показуємо локальний стан джерела з CTA «Відкрити модуль» або «Спробувати ще раз».

Поточні preference-прапорці `showFizrukInCalendar` та `showFinykSubscriptionsInCalendar` у `RoutineState.prefs` можна використати як основу, але onboarding має зробити їх зрозумілими для користувача.

## 7. Не входить у MVP

- довільні користувацькі події та повноцінний task manager;
- підтвердження фінансових або тренувальних подій безпосередньо в Routine;
- новий кросмодульний backend-агрегатор, якщо наявних локальних mirror/domain helpers достатньо;
- автоматичне планування вільного часу;
- AI-рекомендації та predictive scheduling;
- mobile parity у цій самій UX-ітерації, якщо це збільшує ризик для web MVP.

## 8. Технічна привʼязка до поточного коду

Перший web slice має розвивати наявні boundaries:

- `apps/web/src/modules/routine/RoutineApp.tsx` — composition root;
- `apps/web/src/modules/routine/RoutineTimeline.tsx` — shell і loading/error states;
- `apps/web/src/modules/routine/components/RoutineCalendarPanel.tsx` — timeline, day/week/month та джерельні фільтри;
- `apps/web/src/modules/routine/context/RoutineCalendarContext.tsx` — data/actions contract;
- `apps/web/src/modules/routine/useRoutineAppState.ts` — orchestration та persisted preferences;
- `apps/web/src/modules/routine/lib/hubCalendarAggregate.ts` — агрегування подій;
- `apps/web/src/modules/routine/lib/finykSubscriptionCalendar.ts` — Finyk subscription projection;
- `packages/routine-domain/src/` — чисті типи, календарна логіка, streak і date-key invariants.

Потрібно уникати розширення `RoutineCalendarPanel.tsx` до нового моноліту: timeline item, source setup/onboarding і empty states мають бути окремими view-компонентами або чистими helpers.

## 9. Acceptance criteria для MVP

- [ ] Новий користувач бачить onboarding перед timeline і сам обирає Finyk/Fizruk.
- [ ] Після onboarding Routine відкривається на timeline поточного Kyiv дня.
- [ ] Власні звички та підключені модульні події відображаються в єдиному порядку.
- [ ] Події без часу відображаються в окремому блоці.
- [ ] Власну звичку можна виконати з timeline або відкрити в detail.
- [ ] Кросмодульна подія не змінюється з Routine; натискання веде у джерельний модуль.
- [ ] Вимкнення джерела прибирає його події з Routine без видалення даних у джерелі.
- [ ] Є loading, empty, unavailable-source та error стани.
- [ ] Day є default, week/month залишаються доступними для огляду.
- [ ] Є unit-тести для сортування, джерельних toggles, untimed-групи та Kyiv day boundary.
- [ ] Є component/E2E smoke для першого запуску, переходу в джерельний модуль і швидкого виконання звички.

## 10. Метрики успіху

Після реалізації варто вимірювати:

- завершення onboarding Routine;
- частку користувачів, які підключили хоча б одне джерело;
- відкриття Routine → взаємодію з timeline;
- швидке виконання звички з timeline;
- переходи з Finyk/Fizruk-подій у джерельний модуль;
- повернення в Routine протягом 7 днів;
- частку користувачів, які вимикають окреме джерело через шум.

## 11. Відкриті рішення для наступної сесії

1. Які саме фінансові події входять у перший набір: лише підписки/регулярні платежі чи також ручні заплановані платежі?
2. Чи мають тренування мати час у Fizruk, чи всі безчасові тренування спочатку потрапляють у блок «Без часу»?
3. Який default для завершених подій: завжди згорнутий чи запамʼятовувати вибір користувача?
4. Чи потрібні окремі source filters на day view, якщо onboarding вже обмежив джерела?
5. Який мінімальний набір статистики має залишитися на вкладці Stats після появи timeline-first landing?
