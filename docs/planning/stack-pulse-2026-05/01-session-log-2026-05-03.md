## Що поїхало в production

### PR-03 / C3 — bcrypt 72-byte limit ([#1550](https://github.com/Skords-01/Sergeant/pull/1550))

Bcrypt мовчки відрізає вхід на 72 байти. До цього PR `MAX_PASSWORD_LENGTH=128`
у `apps/server/src/env.ts` створював security theatre — користувач вводив
80-символьний пароль, перші 72 байти переходили у hash, решта мовчки
відкидалася. Дві паролі, що різняться лише після 72-го байта, давали один і
той же hash.

**Зміни:**

- `MAX_PASSWORD_LENGTH=72` як default (zod cap у `env/env.ts`, parseInt cap у
  legacy `env.ts`, оновлений `.env.example`).
- ADR-0042 — `docs/adr/0042-password-hashing-strategy.md` з планом міграції
  на Argon2id (Option B), якщо колись захочемо переїхати з bcrypt.

### PR-02 / C2 — rate-limit fail-closed на `/api/auth/*` ([#1552](https://github.com/Skords-01/Sergeant/pull/1552))

`apps/server/src/http/rateLimit.ts` мав caching-fallback chain
Redis → in-memory. На multi-replica deploy (Railway = 3 replicas) при відмові
Redis кожна репліка вела свій per-process bucket, тож attacker фактично
отримував `replicas × limit` спроб на window. Для credential-stuffing-у це
підіймало `5/15min` ліміт до `15/15min`, що скасовувало сенс рейт-ліміту.

Тим часом codebase встиг ввести **другий** fallback — Postgres
(`rate_limit_buckets`). Тож актуальний chain був Redis → Postgres → in-memory,
і fail-open проблема залишалась — тільки коли впадуть **обидва** Redis і
Postgres.

**Зміни:**

- `RATE_LIMIT_FAIL_CLOSED_AUTH=true` (default) у обох env-modul-ях.
- `failMode: "open" | "closed"` у `RateLimitOptions`.
- `tryCheckRateLimitPg()` — sentinel pattern: повертає `null` при collapse,
  щоб caller міг розрізнити «PG спрацював і дав ok» від «PG впав».
- `apps/server/src/http/authMiddleware.ts` — wires `failMode: "closed"` для
  `/sign-in`, `/sign-up`, `/forget-password`, `/reset-password`.
- При обох-down + closed: 503 + `Retry-After: 5s` + `code: RATE_LIMIT_UNAVAILABLE`.
- Counter `rate_limit_degraded_total{key,mode}` — степ degraded-ит-mode
  стає alert-able, не silent.
- 6 нових тестів у `apps/server/src/http/rateLimit.test.ts`.
- Runbook `docs/security/rate-limit-failure-mode.md` (threat-model +
  configuration matrix + monitoring).

### PR-15 / H9 — `AI_QUOTA_DISABLED` hard-block у production ([#1567](https://github.com/Skords-01/Sergeant/pull/1567))

`AI_QUOTA_DISABLED=1` — fail-open kill-switch на per-user / per-IP AI-квоту.
У CI це sensible (e2e ганяють реальний Anthropic без burning-у user quota).
У production — копіпаста зі staging або helm-typo дає клієнтам unlimited
Anthropic budget. Попередній safeguard — `logger.warn` на module-import — був
невидимий серед startup-warnings.

**Зміни:**

- `AI_QUOTA_DISABLED` тепер typed boolean (zod `enum([true,false,1,0,""])
.default("false").transform(...)`), default `false`.
- `assertStartupEnv()` хард-кидає, якщо `isProduction && env.AI_QUOTA_DISABLED`.
  `isProduction` = `NODE_ENV=production` ∨ `RAILWAY_ENVIRONMENT` ∨
  `RAILWAY_SERVICE_NAME` (Railway boots без `NODE_ENV=production` явно).
- `aiQuota.ts` `isAiQuotaDisabled()` приймає і `1`, і `true` — синхронно з
  typed env. Старий module-load `logger.warn` зник, бо startup hard-block
  його замінив.
- `.github/workflows/extended-e2e.yml` — `NODE_ENV: test` явно, щоб workflow
  продовжив бути bootable після hard-block-у.
- `docs/security/ai-quota-kill-switch.md` — runbook (чому існує, чому prod
  заблокований, allowed-configurations matrix, test coverage, ops guidance).
- 9-кейсова матриця у `apps/server/src/env/__tests__/assertStartupEnv.test.ts`.
