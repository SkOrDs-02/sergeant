## 1. 🔴 BLOCKER — Web bootstrap дохне (`node:fs` у клієнтському бандлі)

**Файл:** [`apps/web/src/core/db/kvStoreBoot.ts`](../../apps/web/src/core/db/kvStoreBoot.ts) (рядок 41)
**Симптом:** перехід на будь-який URL (`/welcome?demo=1`, `/`, etc.) дає білу сторінку. `<div id="root"></div>` залишається порожнім. Error-boundary не встигає рендеритись, бо crash станеться на стадії модульного парсу.

### Що бачиться у DevTools

```
Uncaught Error: Module "node:fs" has been externalized for browser compatibility.
Cannot access "node:fs.promises" in client code.
  at Object.get (__vite-browser-external:node:fs:3:11)
  at files.ts:1:32
```

Vite-сервер дзеркалить це у логах:

```
[Unhandled error] Error: Module "node:fs" has been externalized for browser compatibility.
  Cannot access "node:fs.promises" in client code.
 > Object.get ../../../../../../@id/__vite-browser-external:node:fs:3:11
 > ../../packages/db-schema/src/migrate/files.ts:1:31
```

DOM (порожній root):

```html
<body>
  <div id="root"></div>
  <!-- React ніколи не отрендерив, бо crash станеться на стадії модульного парсу -->
</body>
```

### Кореневa причина

```ts
// apps/web/src/core/db/kvStoreBoot.ts:32–41
import {
  KV_STORE_CLIENT_MIGRATIONS,
  KV_STORE_MIGRATIONS_TABLE,
} from "@sergeant/db-schema/sqlite";
import { kvStore } from "@sergeant/db-schema/sqlite";
import {
  createSqliteAdapter,
  type SqliteMigrationClient,
} from "@sergeant/db-schema/migrate/sqlite";
import { runMigrations } from "@sergeant/db-schema/migrate"; // ← BUG
```

Імпорт іде з umbrella-сабпасу `@sergeant/db-schema/migrate`, який у `packages/db-schema/src/migrate/index.ts` re-експортить **усе**, включно з `loadMigrationFiles` із `./files.ts` — а той топ-левел імпортить `node:fs/promises` + `node:path`:

```ts
// packages/db-schema/src/migrate/files.ts:1–2
import { promises as fs } from "node:fs";
import path from "node:path";
```

Бандлер тягне за гілкою → у браузері `loadMigrationFiles` намагається доторкнутися `node:fs.promises` → виняток на boot.

### Це **не нова** пастка — її уже знає весь репозиторій

Інші браузерні споживачі ([`apps/web/src/modules/routine/lib/clientMigrate.ts`](../../apps/web/src/modules/routine/lib/clientMigrate.ts), [`apps/web/src/modules/fizruk/lib/clientMigrate.ts`](../../apps/web/src/modules/fizruk/lib/clientMigrate.ts)) вже мігрували на безпечний саб-сегмент і навіть лишили docstring-warning:

```ts
// apps/web/src/modules/routine/lib/clientMigrate.ts:5–11
// Import the runner from the dedicated `./migrate/runner` sub-path
// rather than `./migrate`. The umbrella `./migrate` entry re-exports
// `loadMigrationFiles` from `./files.js`, which top-level imports
// `node:fs` / `node:path` and breaks Vite's browser bundle when the
// SPIKE library lands in the production graph (see PR #1378
// follow-up). The runner itself is dialect- and platform-free.
import { runMigrations } from "@sergeant/db-schema/migrate/runner";
```

…а от свіжий PR #062 (`kvStoreBoot.ts`) цей урок проігнорував.

## 3. 🟠 High — `apps/web` Vitest: 3 фейли в `useAppLock`

[`apps/web/src/core/security/useAppLock.test.ts`](../../apps/web/src/core/security/useAppLock.test.ts):

```
❯ src/core/security/useAppLock.test.ts (9 tests | 3 failed) 61ms
  × returns 'locked' when flag enabled and PIN is stored (6ms)
  × unlock() with correct PIN transitions to 'idle' and returns true (2ms)
  × unlock() with wrong PIN keeps 'locked' and returns false (3ms)
```

### Реальний traceback (вийняв через `pnpm vitest run … --reporter=verbose`)

```
TypeError: Failed to normalize algorithm:
  'salt' of 'Pbkdf2Params' (passed algorithm) is not instance of
  ArrayBuffer, Buffer, TypedArray, or DataView.
  ❯ deriveHash src/core/security/lockStorage.ts:40:36
  ❯ Module.savePinHash src/core/security/lockStorage.ts:62:16
```

### Конкретна проблема

```ts
// apps/web/src/core/security/lockStorage.ts:40–49
const bits = await crypto.subtle.deriveBits(
  {
    name: "PBKDF2",
    salt: salt.buffer as ArrayBuffer, // ← B
    iterations: 200_000,
    hash: "SHA-256",
  },
  keyMaterial,
  256,
);
```

`salt: salt.buffer as ArrayBuffer` — на Node 22 + jsdom + Vitest, `Uint8Array.buffer` віддає об'єкт із "чужого" realm-у, тому `instanceof ArrayBuffer` всередині WebCrypto false-ить. Якщо передати `salt` напряму (TypedArray), рантайм приймає.

### Це _не_ регресія user-facing-логіки

- `useAppLock.ts` сам по собі OK.
- В **продакшен-Chromium** код працює (`crypto.subtle.deriveBits` з `Uint8Array.buffer` приймається V8/Blink).
- Падає лише **тест** через node-jsdom особливість.
- Але **CI блокує мерж** через цю помилку, тому фактично — баг гілки, але не баг застосунку.

### Що це говорить нам про процес

Тест PR-1a UX-roast (`63c2858f feat(web): add app-lock base PIN flow`) додано **сьогодні** (`2026-05-07 12:54 UTC`, автор Claude AI). Він ніколи не проганявся локально на Node 22. Це натяк на процес: PR-и UX-roast серії мерджаться без локального `pnpm test`.

**Рекомендація:** простий fix потім (`salt: salt as BufferSource`), але важливіше — додати в playbook UX-roast явну вимогу `pnpm --filter @sergeant/web test` перед мерджем.

## 8. 🟡 Latent — ще 5 однакових bombs у mobile (дзеркало #1)

Скан репо показав, що umbrella-імпорт `@sergeant/db-schema/migrate` тягне ще **п'ять** мобільних споживачів — не один:

```
apps/web/src/core/db/kvStoreBoot.ts:41          ← ЗЛОВИВ Vite (наш блокер)
apps/mobile/src/core/db/kvStoreBoot.ts:39       ← дзеркало того ж самого
apps/mobile/src/modules/finyk/lib/clientMigrate.ts:5
apps/mobile/src/modules/fizruk/lib/clientMigrate.ts:5
apps/mobile/src/modules/nutrition/lib/clientMigrate.ts:5
apps/mobile/src/modules/routine/lib/clientMigrate.ts:5
```

Web-сторона вже мігрувала всі 4 module-`clientMigrate` на безпечний `/migrate/runner` (Routine навіть із докладним коментарем-попередженням). **Mobile сторона — ні.**

Чому Metro поки не "вибухнув" так само, як Vite?

- Metro за замовчуванням **не резолвить** `node:` префікс — він транспілить `node:fs` як missing module **під час bundling**, а не runtime.
- Імпорт `loadMigrationFiles` від `files.ts` теоретично провалиться як unresolved, але Metro tree-shake-ить unused exports з re-export'ів **на рівні файлів** (named-export hoisting). Тому **поки** ніхто не зове `loadMigrationFiles` рантаймно з мобільного, проблема latent.
- Якщо хтось додасть code-path, що **тягне** будь-який експорт із `./files.ts` транзитивно — все, що exporting через umbrella, втрапить у бандл і розвалить production build. Це міна в підлозі.

**Висновок:** мобільні 5 imports — не активний краш, але **fragility debt**. Привести усі 6 (5 mobile + 1 web) до `/migrate/runner` — однорядкові правки. Це робить рефакторинг "drop the umbrella" із §10 реалістичним.

## 9. Чому процес ревʼю не зловив блокер #1

PR #062 / #063 / #064 пройшли всі чотири стадії (lint / typecheck / unit-test / merge), не зловивши блокер. Чому?

## A. Production-помилки з Sentry (додано 2026-05-07 пізно)

> Користувач відкрив `dima-dk.sentry.io/issues` та надіслав скріншот фіду `is unresolved · 14D`. Знайдено два чисто-продові регресії, що не покриваються типечеком/лінтом/юніт-тестами і **не пов'язані** з web-blocker'ом §1 (їхні події старші за 24h, сягають **2 дні** — ще до ввімкнення PR #064).

### A1 — `SQLITE_ERROR: no such table: sync_op_outbox` (web prod, 🔴 P0)

**Де:** SERGEANT-WEB-A / -5 / -6 / -7 / -8 / -9 (сім окремих груп = 11 events / 9 unique users) — найновіше **17 секунд** до зйомки скріна, найстаріше **2 дні**, лінія тренду по всіх «New».
**Канонічний меседж:** `SQLITE_ERROR: sqlite3 result code 1: no such table: sync_op_outbox`.

**Що це за таблиця.** Створюється у клієнтській SQLite-міграції [`packages/db-schema/src/sqlite/migrations/index.ts:73`](../../packages/db-schema/src/sqlite/migrations/index.ts) (з пакета `001_routine_spike.sql`, який і досі живе як «нульовий» SPIKE для модуля Routine). Це **SQLite-only** таблиця — для серверного PG-шляху `sync_op_outbox` не існує, тому помилка спрацьовує винятково у браузерному WASM-SQLite.

**Звідки запит.** Єдиний споживач у `apps/web/src` —
[`apps/web/src/modules/routine/lib/dualWrite/adapter.ts:40`](../../apps/web/src/modules/routine/lib/dualWrite/adapter.ts) (Routine dual-write pipeline) → `outboxEnqueue` → `sync_op_outbox`. Таблицю мав би створити `runMigrations({ migrations: ROUTINE_SQLITE_BUNDLED_MIGRATIONS })` під час `kvStoreBoot()`/еквівалент Routine.

**Гіпотеза root-cause** (потребує одного підтверджуючого запиту у Sentry — `tags.user.id` та `breadcrumbs` з фіду):

1. **Best-fit:** події **передують** P0-блокеру §1 (oldest 2d, тобто до `90c5626a` 2026-05-07 12:52 UTC), отже kvStoreBoot тоді **запускався**, але `runMigrations` для Routine SQLite не дочекався (race / помилка на etape build PR-040 retry-policy migration). PR #040 (`002_sync_op_outbox_retry.sql`) переробляє таблицю через `ALTER ... RENAME → CREATE → INSERT SELECT → DROP legacy`. Якщо `INSERT SELECT` упало, таблиця залишається у стані `sync_op_outbox_legacy` (без активного `sync_op_outbox`) → `outboxEnqueue` бачить «no such table». Це **відкоментовано як ризик** прямо в коді:
   ```ts
   // packages/db-schema/src/sqlite/migrations/index.ts:131
   * The whole thing runs inside the per-migration `BEGIN`/`COMMIT` the
   * runner installs around each file, so an interrupted migration leaves
   * the legacy table intact. — # коментар лишений у файлі
   ```
2. **Альтернатива:** користувач відкрив старий tab → `wasm-sqlite` ініціалізував DB до того, як міграційний раннер встиг наздогнати; потім запит `outboxEnqueue` вилетів. Тоді на rerender тред, що мав почати міграції, валиться раніше (наприклад через P0 §1 — після 2026-05-07 12:52 UTC).

**Що робити (без коду — ідея для PR):**

- Додати `sync_op_outbox` smoke-перевірку в `routineDualWriteAdapter.outboxEnqueue` (якщо `SELECT name FROM sqlite_master WHERE name='sync_op_outbox'` нема — запустити `runMigrations` синхронно ще раз, або кинути типизовану `SyncSetupRequiredError` зі скиданням dual-write на простий direct-write).
- Логнути у Sentry breadcrumbs `migrations.applied` set і `__migrations` row count, щоб у наступному звіті відрізнити «runner not started» vs «runner crashed mid-flight».
- Перевірити, що `runMigrations` на web дійсно запускає 001/002/003 (можливо умбрелла-фікс §1 тимчасово вимкнув виклик — треба перевірити після рестору). У такому разі ця помилка зникне «безкоштовно» з §1-фіксом, але користувачі що вже у поганому стані залишаться зламаними і мають ручний reset path (видалення local OPFS DB).

**Не root cause** (відкинуто): «таблицю перейменували» — `git log --diff-filter=R -- packages/db-schema/src/sqlite/migrations/index.ts` нічого не показує.

### A2 — `column "created_at" does not exist` POST `/api/internal/openclaw/query` (api prod, 🟠 P1, **Unhandled**)

**Де:** SERGEANT-API-9, 13 events / **0 distinct users** (значить це bot/agent traffic, не end-user) за 4d. Helmet «Unhandled» означає 5xx Express middleware чейн.
**Канонічний меседж:** `error: column "created_at" does not exist`.

**Звідки запит.** [`apps/server/src/routes/internal/openclaw.ts:340-360`](../../apps/server/src/routes/internal/openclaw.ts) — `r.post("/api/internal/openclaw/query", …)` приймає вільний `sql` з body, валідує лише `min(1).max(8000)` + allowlist `SELECT/WITH only` + table allowlist. Power-tool для Sergeant Console DM-handler'а (LLM пише SQL → сервер виконує).

**Чому валиться.** Реальні таблиці openclaw мають `*_at` колонки **з різними іменами**:

```sql
-- apps/server/src/migrations/028_openclaw.sql
CREATE TABLE openclaw_decisions (
  decided_at TIMESTAMPTZ NOT NULL DEFAULT NOW(),  -- НЕ created_at
  …
);
CREATE TABLE openclaw_invocations (
  invoked_at TIMESTAMPTZ NOT NULL DEFAULT NOW(),  -- НЕ created_at
  …
);
-- apps/server/src/migrations/030_openclaw_write_audit.sql
CREATE TABLE openclaw_write_audit (
  recorded_at TIMESTAMPTZ NOT NULL DEFAULT NOW(),  -- НЕ created_at
  …
);
```

Жодна openclaw-таблиця не має колонки `created_at`. LLM (chat assistant у Console) має застаріле уявлення про схему і генерує:

```sql
SELECT … FROM openclaw_invocations ORDER BY created_at DESC LIMIT 20;
```

Сервер пускає такий SQL у `pg.Pool.query()` — Postgres повертає `error: column "created_at" does not exist`, що в `queryAppDb` **не оброблено** ([`apps/server/src/modules/openclaw/tools.ts:208-300`](../../apps/server/src/modules/openclaw/tools.ts)) → asyncHandler катає у Express → 500 Unhandled у Sentry.

**Як народилася регресія.**

```
$ git log --oneline -- apps/server/src/migrations/028_openclaw.sql
a1f0aa90 feat(server): add openclaw decisions/invocations tables  (≈2026-04-12)
```

PR #028 одразу заклав `decided_at`/`invoked_at` як кращі імена за «common-noun» `created_at`. Cofounder/Console агент (LLM-prompt у `tools/openclaw/src/agents/openclaw.ts`) має у contexts cheat-sheet, що **посилається на правильні стовпці** (рядок 126 згадує allowlist), але system-prompt очевидно недостатньо явний — LLM «галюцинує» універсальне `created_at`.

**Що робити (без коду — ідея для PR):**

- **Server-side guard:** обернути `queryAppDb` у `try/catch` + повертати `400 { error: "schema_error", message }` замість 500. Поточний catch ловить тільки `OpenClawAllowlistError`. Унеможливить «Unhandled» Sentry-shouting і дасть LLM-у самозалатати запит з рідапту (агент же вже бачить `error.message` у tool-call response).
- **Prompt-side guard:** додати у `tools/openclaw/src/agents/openclaw.ts` `query_app_db` system-instructions блок з табличним переліком стовпців (`openclaw_invocations.invoked_at`, `openclaw_decisions.decided_at`, `openclaw_write_audit.recorded_at`) — щоб LLM не вгадував.
- **Optional view shim:** створити PG-вʼюшку `openclaw_*_view` з колонкою `created_at AS invoked_at/decided_at/recorded_at`. Перевага: backwards-compat для будь-яких legacy скриптів. Недолік: маскує справжню проблему prompt'а.

**Чому це 🟠 а не 🔴:** не блокує end-user'ів (engagement bot tools), але заваджає сафтай-неті спостережуваності — кожне «Unhandled» — це ілюзія справжнього end-user 500-ка.
