**Скоуп:** перший досвід після установки до ~7-го дня. **Онбординг (welcome / wizard / permissions-prompt) винесений за дужки** — не чіпаємо (це домен `2026-05-03-ftux-onboarding-roast.md`).

**Метод:** статичний код-рев'ю репо `Skords-01/Sergeant` (web-app, `apps/web/src/**`) + перехресна перевірка з i18n-каталогом (`shared/i18n/uk.ts`), компонентами хабу/модулів, settings, error-boundary, форм аутентифікації, чату.

**Формат:** P0/P1/P2 пріоритети + явні дії «додати / змінити / прибрати». Конкретні файли/рядки в посиланнях.

**TL;DR — топ-7 болів:**

1. **Нема app-lock / PIN**, але всередині Mono-токен, баланси, "Пам'ять" асистента, медичні дані. Найбільший trust-risk на FTUX (P0).
2. **Назви головної вкладки в кожному модулі різні** ("Огляд" / "Сьогодні" / "Календар"). Хаб каже "Головна". IA-розхитаність (P0).
3. **Налаштування не там, де очікують.** У модулі немає шестерні — налаштування Фініка живуть у вкладці хабу `Налаштування → Фінік`. Користувач, який сидить у модулі, цього не знайде (P0).
4. **Logout захований**, тоді як видалення акаунту виставлено окремою collapsible-секцією у `Профілі`. Інверсія приоритетів (P1, частково виправлено в `GeneralSection`, але дублюється і плутає).
5. **Англомовний "blocked / Retry"** в `OfflineBanner` коли всюди UA — копірайт-баг (P1).
6. **Резервна копія описана як «Фінік, Фізрук, Рутина, останній модуль»** — пропущено Харчування і додано загадковий «останній модуль» (P1, копірайт-баг).
7. **Відсутня уніфікована empty-state-бібліотека на всіх 4 модулях** — десь є illustration+CTA, десь голий "Немає даних". Перший день виглядає по-різному (P1).

## 0. Карта екранів після інстала (без онбордингу)

```
[install]
   │
   ├─ /welcome           (онбординг — поза скоупом)
   │
   └─ /                  (Hub)
        ├─ Tab: Головна       → HubDashboard
        │                       ├─ Hero (FirstActionHeroCard | SoftAuthPromptCard | TodayFocusCard)
        │                       ├─ Bento 2×2 (Фінік / Фізрук / Рутина / Харчування)
        │                       ├─ Insights, Daily Nudge, Weekly Digest, Re-engagement
        │                       └─ FAB → /chat
        ├─ Tab: Звіти          → HubReports        (з'являється після 1-го запису)
        ├─ Tab: Профіль|Увійти → ProfilePage | AuthPage
        └─ Tab: Налаштування   → HubSettingsPage
                                  └─ search + 13 секцій: Dashboard, General, Notifications,
                                     AI Digest, Assistant Catalogue, Routine, Fizruk, Finyk,
                                     Nutrition, PWA, Data Export, Experimental
        ├─ Header: Search (⌘K), DarkModeToggle, [Sign-in icon]
        └─ Banners: Update / Install / iOS-install / OfflineBanner

  Standalone routes:
    /sign-in, /chat, /assistant, /pricing, /design, /reset-password
    Modules: /finyk, /fizruk, /nutrition, /routine
```

Усе керується одним правилом: **під час FTUX-сесії (до першого «реального» запису) Reports/банери приховуються — single-signal-FTUX**. Це добре. Але **після першого запису** все одночасно вибухає (Reports tab з bounce + toast, install/iOS-banner/Update banner reactivated, Re-engagement card можливо), і дашборд починає виглядати галасливим.

## 1. НАВІГАЦІЯ та інформаційна архітектура

### 1.1 ❌ Назви головної вкладки в кожному модулі різні (P0)

Це наочно видно з модуль-нав-файлів:

- Хаб: `Головна / Звіти / Профіль / Налаштування`
- Фінік: `Огляд / Операції / Планування / Аналітика / Активи` — `apps/web/src/modules/finyk/components/finykNav.tsx`
- Фізрук: `Сьогодні / Тренування / План / Прогрес і заміри / Моє тіло` — `apps/web/src/modules/fizruk/shell/fizrukNav.tsx`
- Харчування: `Огляд / Склад / Журнал / Меню` — `apps/web/src/modules/nutrition/components/NutritionBottomNav.tsx`
- Рутина: `Календар / Статистика / …` — `apps/web/src/modules/routine/components/RoutineBottomNav.tsx`

Перша вкладка (semantically — landing) називається **«Огляд»**, **«Сьогодні»**, **«Огляд»**, **«Календар»**. Користувач, який тільки-но засвоїв слово «Головна» в хабі, відкриває Фізрук і має згадувати, що «головна» там називається «Сьогодні».

**Зміна:**

- Привести **всі модулі** до однієї назви першої вкладки. Найкращий кандидат — **«Огляд»** (нейтральне, описує overview-screen). У Фізруку «Сьогодні» — це специфічна частина, а реальний overview це поточна програма + статистика → нехай теж буде «Огляд», а «Сьогодні» — підтаб.
- У Рутині перша вкладка має сенс як «Огляд», з календарем як основним візуалом. «Календар» — це інструмент, а не landing-сурфейс.

### 1.2 ❌ «Налаштування модуля» поза модулем (P0)

Користувач у Фініку хоче змінити свої кастомні категорії або відключити Mono. Шукає шестерню в модулі — її **немає** в `FizrukHeader`/`NutritionHeader`/finyk overview header. Потрібно:

1. Закрити модуль («Назад до хабу»)
2. Перейти на вкладку «Налаштування»
3. Знайти секцію «Фінік» серед 13-ти
4. Розгорнути її

Це 4 кліки замість 1. **`HubSettingsPage` має дуже хороший пошук**, але user не знає, що шукати в хабі, коли проблема в модулі.

**Додати:**

- У header кожного модуля — іконку шестерні, яка відкриває хаб-settings із `#settings-finyk` (deep-link уже підтримується через hash-роутинг — `apps/web/src/core/hub/HubSettingsPage.tsx`).
- Альтернативно: рендерити «mini-settings» прямо як sheet з модуля, без виходу.

### 1.3 ❌ Backbutton-інконсистентність (P1)

`HubBottomNav` має `Профіль/Увійти` — лейбл міняється залежно від auth. Це добре. Але:

- У `FinykLoginScreen` бек-кнопка може мати кастомний лейбл або не показуватись (`apps/web/src/modules/finyk/components/FinykLoginScreen.tsx:271`).
- У модулях іноді є кнопка «Назад до хабу», іноді — «Назад» в `ModuleHeader`.
- У `NotFoundPage` — «На головну» з іконкою home.

**Зміна:** єдиний паттерн у `ModuleHeaderBackButton`, лейбл `«До хабу»`. Бек з підекранів модуля → лейбл = назва батьківського табу.

### 1.4 ❌ Reports tab появляється з bounce + toast (P1)

Після першого запису з'являється Reports tab з анімацією bounce і одноразовим toast-ом (`apps/web/src/core/app/HubBottomNav.tsx`, storage key `sergeant.hub.reportsTabRevealedAt`). Це подвійний сигнал на одне і те ж саме явище. Користувач, який щойно проскипав радість «о, я зробив запис», бачить:

1. Анімацію bounce на нижньому табі
2. Toast «нова вкладка з'явилась»
3. Знятий FTUX-suppression → з'являється FAB AI
4. (Можливо) Install banner / iOS banner / Update banner
5. Re-engagement / Soft-auth card на дашборді

Переповнення сигналів. **Прибрати:** Toast про нову вкладку (анімація bounce достатня). У `HubMainContent` є явні правила priority-черги для банерів, але вони все одно конкурують за увагу з hero-карткою.

### 1.5 ❌ ⌘K показує підказку, а на mobile undiscoverable (P2)

Tooltip `«Пошук по всіх модулях (⌘K)»` (`HubHeader.tsx:114`) — корисно, але на mobile shortcut-у нема. Mobile користувач отримує іконку, але не дізнається, що там можна **AI-команди** виконувати (`InlineAiRail` всередині пошуку). Додати mobile-onlу пояснювальний chip нижче пошукового інпуту: `«Запитай асистента або шукай по даних»`.

## 2. КОПІРАЙТ та мікротекст

### 2.1 ❌ Англомовний `blocked / Retry` в OfflineBanner (P1, копі-баг)

`apps/web/src/core/app/OfflineBanner.tsx:41,49`:

```tsx
<span>{`${syncV2DeadLetterCount} blocked`}</span>
…
<button …>Retry</button>
```

Решта банеру українською («Офлайн», «Синхронізація · N в черзі»). Треба «N заблоковано» і «Повторити». Окремо: «blocked» як саме слово user не зрозуміє — треба «N не синхронізовано» / «Спробувати ще раз».

### 2.2 ❌ DataExport — некоректний перелік модулів (P1, копі-баг)

`apps/web/src/core/hub/HubBackupPanel.tsx:53`:

> «Резервна копія всього Hub (Фінік, Фізрук, Рутина, **останній модуль**).»

«Останній модуль» — це або плейсхолдер, або помилка. Має бути **Харчування**. Користувач, який бачить «не названо» свій модуль, припускає, що його дані не бекапляться.

**Змінити** на: «Резервна копія всього Hub (Фінік, Фізрук, Рутина, Харчування) у JSON-файл.»

### 2.3 ❌ «Подивитись tour» vs «Перезапустити онбординг» — два різних інструменти, неочевидна різниця (P1)

`apps/web/src/core/settings/GeneralSection.tsx:51-83`. Обидві кнопки в одному під-блоці, без явного контрасту візуально. User читає текст:

> «Подивитись tour — побачити вітальний екран ще раз без скидання твого стану. Перезапуск не видаляє твої дані — повертає вітальний екран і підказки першого запуску.»

Тобто **обидві не видаляють дані**? У чому тоді різниця? Має бути:

- "Переглянути екскурсію" — вторинна дія (зараз `ghost`)
- "Скинути prefs FTUX" — пояснити, що «крок-каунтер скинеться, vibe-picks перевиберуться», а **не** «дані модулів». Вже зараз ці кнопки сидять впритул, і назви двох не рівнозначні (одна каже про стан, друга — каже «перезапустити»). Користувач не розрізнить «онбординг-стан» vs «його дані».

### 2.4 ❌ Слова «Фінік / Фізрук» — чарівні бренди, але непрозорі для нового user (P1)

Перший раз бачачи «Фінік», користувач не знає, що це фінанси. **На модульних картках на дашборді** (HubDashboard bento) є лише іконка + бренд-нейм. Без сабтайтлу типу «Фінанси · Mono / витрати / бюджети» новий user буде гадати.

**Додати:** `subtitle` під назвою модуля на bento-картці, що описує домен:

- Фінік → «Фінанси та бюджети»
- Фізрук → «Тренування та тіло»
- Рутина → «Звички та трекери»
- Харчування → «Їжа, склад, рецепти»

(У `WelcomeScreen` ці підписи тимчасово вже є як `metric/sub` для blur-peek, але це fake-дані; на реальному дашборді одразу пропадає.)

### 2.5 ❌ «AI-діагностика SW» — UX тестового інженера, не користувача (P1)

`apps/web/src/core/settings/PWASection.tsx`:

- `«Діагностика SW»` → «Не вдалося отримати діагностику SW», `«SW-діагностика виведена в консоль»`

Звичайний user не знає, що таке SW. І не дивиться console. Краще:

- Перейменувати в «Технічна перевірка PWA»
- Прибрати кнопку «Діагностика SW» з UI зовсім, **залишити тільки в Experimental або ховати під «Розгорнути технічні опції»**.

### 2.6 ❌ "Зачекайте…" vs "Підключення…" vs "Виходимо…" — стилістичний хаос (P2)

i18n-каталог має `status.loading`, `status.updating`, `actions.tryAgain`. Але:

- AuthPage register submit → `«Зачекайте…»` (нестандартно)
- FinykLoginScreen → `«Підключення…»`
- GeneralSection logout → `«Виходимо…»`
- GeneralSection sync → `«Зберігаємо…»` / `«Завантаження…»`

**Зміна:** уніфікувати під одне з:

- «Зберігаю…» (1-а особа од., як весь tone of voice)
- «Завантажую…»
- «Виходжу…»

`tone of voice` додатку — на «ти», 1-а особа («я») від асистента, але деякі стани в 1-й особі множ. («Зберігаємо», «Виходимо»). Визначити: **завжди «я» або завжди безособово**.

### 2.7 ❌ "Ласкаво просимо!" в AuthPage register success — формальний контраст до решти (P2)

`apps/web/src/core/auth/AuthPage.tsx:302`. Решта застосунка — `«Привіт, X!»`, `«Доброго дня»`. Форма реєстрації завершується урочистим `«Ласкаво просимо!»`. Це не плохо для celebration-modal-а, але ламає голос. Натомість: `«Готово, [name]! Твої дані тепер з тобою на всіх пристроях.»`

### 2.8 ❌ Нагадування про сповіщення — повторний текст (P2)

`NotificationsSection.tsx`:

- Routine toggle: «Без дозволу на сповіщення нагадування не надсилатимуться. Дозволь сповіщення у налаштуваннях браузера.»
- Fizruk toggle: «Без дозволу на сповіщення нагадування не надсилатимуться.»
- Nutrition toggle: «Без дозволу на сповіщення нагадування не надсилатимуться.»

Три однакові toast. Краще — **перевіряти ОДИН раз** (на найпершому toggle), а далі дозвіл уже видано. Або винести як loop — `requestRoutineNotificationPermission()` уже centralized.

## 3. EMPTY STATES

### 3.1 ❌ Inline `«Немає даних»` в HubReports замість стандартизованого EmptyState (P1)

`apps/web/src/core/hub/HubReports.tsx:84-90`:

```tsx
if (!hasData) {
  return (
    <div className="h-24 flex items-center justify-center text-xs text-muted">
      Немає даних
    </div>
  );
}
```

Стандартний `EmptyState` (`shared/components/ui/EmptyState.tsx`) з ілюстрацією, описом, CTA-кнопкою — повз. Result: на новому user (1 день) розділ Reports — це купа крихітних `«Немає даних»` без жодного guidance. Звіти за тиждень будуть порожні до 7-го дня — і нічого не пропонує що з цим робити.

**Зміна:** замінити inline-empty в кожній StatCard на:

```
illustration: барчарт-skeleton-illustration
title: "Поки даних мало"
description: "Звіт оживе після 3-х записів цього тижня"
hint / action: "Додати запис" → відкриває Quick-Add Sheet
```

### 3.2 ❌ Module bento-карток у FTUX-режимі — статичні fake-числа (P1)

`HubDashboard` рендерить bento-картки модулів навіть до першого запису (хоч hero вище — FirstActionHeroCard). На картках — нулі або плейсхолдери, що візуально читаються як «нічого не працює». Потрібно або:

- На FTUX-режимі — **показувати «Що тут буде»** (mini-illustration + 1 рядок: «Сюди ляже твій баланс і твої витрати за тиждень»)
- Або клікабельні картки з overlay-CTA («Спробувати»)

### 3.3 ❌ Пустий чат AI — нема шаблонних запитів видимо до першого scroll (P2)

`HubChatBody` рендерить `aria-live="polite"` div з повідомленнями. На пустому чаті — нічого. `ChatQuickActions` живуть в `HubChatComposer` нижче. На mobile-клавіатурі вони можуть бути приховані за keyboard inset. **Додати** на пустий-стан рекомендовані запити "як використовувати асистента" вище композера або як 3-4 chip suggestions посеред порожнього body.

### 3.4 ❌ Памʼять (MemoryBank) у Profile — collapsed by default, без preview (P2)

`ProfilePage.tsx:62`: `<MemoryBankSection />` всередині `<CollapsibleSection title="Пам'ять" defaultOpen={false} collapsedSubtitle="Що асистент знає про тебе"/>`. У FTUX там має бути «AI ще не знає про тебе нічого» з іконкою + CTA «Розкажи асистенту про себе». Зараз це гола колапсована секція з натяком.

## 4. LOADING / ERROR / EDGE CASES

### 4.1 ❌ Skeleton-loaders нерівномірні: десь є, десь нема (P1)

`apps/web/src/modules/nutrition/NutritionApp.tsx` імпортує `SkeletonMealCard, SkeletonText, Skeleton`. У Фініку — `Skeleton` з `apps/web/src/modules/finyk/pages/Overview.tsx` (через `@shared/components/ui/Skeleton`). Але:

- HubReports — нема skeleton-у; cтоп-картки рендеряться з `«Немає даних»` одразу.
- Profile sections — нема skeleton-у (sessions list); рендериться без preview.
- Hub Dashboard — Bento картки мерехтять у момент гідрації.

**Додати:** уніфікований skeleton для:

- Reports stat cards (4 cards × 1 chart)
- Profile.sessions list (2-3 fake rows)
- Hub bento (4 placeholder tiles)

### 4.2 ❌ ModuleErrorBoundary показує «Помилка в модулі», але user не знає кому скаржитись (P1)

`apps/web/src/core/ModuleErrorBoundary.tsx` + `messages.errors.generic.moduleFailed`. Кнопки: `Спробувати ще` / `До вибору модуля`. Але немає:

- **Request-ID для копіювання** (root `ErrorBoundary` має `extractRequestId` + `copyRequestId`, а module-rebinding — ні).
- **Send report** кнопки.

**Додати** в `ModuleErrorBoundary` ту ж логіку — щоб з модуля юзер міг скопіювати помилку і скинути тобі.

### 4.3 ❌ Sync error toast — не пояснює, що саме не пішло (P1)

`PullToRefresh` wrapper в `HubMainContent`. На mobile — touch-gesture. На desktop — нічого: треба додати кнопку refresh в header (зараз тільки tooltip над кнопкою «Sync» в settings).

## 8. OFFLINE / SYNC / PERMISSIONS

### 8.1 ❌ OfflineBanner — три розрізнених стану, без єдиного "що робити" (P1)

Поведінка:

- `online && pending===0` → null
- `online && pending>0` → «Синхронізація · N в черзі»
- `!online` → «Офлайн» / «Офлайн · N в черзі»
- `dead-letter > 0` → «N blocked» + Retry (англ., див. §2.1)

User не знає, що **«Офлайн · 3 в черзі»** означає «коли з'явиться інтернет, дані доллються». Додати tooltip або прикрашений expand: тапнути pill → відкрити `Settings → Sync` з actions.

### 8.2 ❌ Permissions — гранулярність є, але **немає глобального overview-стейту** (P1)

`NotificationsSection` показує статус Notification permission. Але:

- **Microphone** — ніде. user натискає mic у чаті → браузер сам запитує. user забуває, що дозволив, і не знаходить, як скасувати.
- **Camera** — те саме (barcode scanner у Nutrition).
- **Geolocation** — якщо колись буде, поки немає.

**Додати** `Settings → Privacy & Permissions` секцію зі списком: Notifications / Microphone / Camera / Location → статус + посилання `«Як скасувати в браузері»` (deep-link на chrome://settings/content для Chrome users).

### 8.3 ❌ Service Worker debug — у Settings, але reload-кнопка ховається разом із "Скинути кеш PWA" (P2)

`PWASection` має 2 ghost-кнопки одного weight. «Скинути кеш PWA» — destructive (reload), але візуально ідентична. Зробити її destructive-tinted або `variant="danger"`.

### 8.4 ❌ Кеш-оверрайди (`requestCloudPull`) можуть мовчазно тригерити refetch під капотом без visual feedback (P2)

`NutritionApp.tsx` юзає `requestCloudPull`. Якщо це тригериться у фоні після toast «Збережено», user бачить spinner десь і не розуміє, чому ще раз refresh. Додати `OfflineBanner` стан `«Доллюємо нові дані»`.

## 11. PRIVACY / TRUST — найбільший P0

### 11.1 ❌ App-lock відсутній (P0)

Sergeant зберігає у LocalStorage:

- Mono-token (плюс PrivatBank, якщо ввімкнено)
- баланси, історію транзакцій
- "Памʼять" про користувача (асистент)

Все доступне будь-кому, хто розблокує телефон. **Жодного PIN/biometric-lock на app**. Це критично для нового user, який тільки інстальнув і думає «ой, тут мої гроші…».

**Додати P0:**

- Settings → Privacy → "Блокування додатка" (PIN 4-6 цифр + biometric якщо доступно).
- Lock-screen при старті, після N хв. неактивності, після перемикання задач.
- Можливість fallback на пароль акаунту.

(Capacitor wrapper = біометрія через `@capacitor-community/biometric-auth` доступна.)

### 11.2 ❌ Привіт-копія в чаті каже "Дані модулів видно офлайн", але не каже **де зберігаються** (P1)

### 11.3 ❌ "Токен Mono зберігається лише у твоєму браузері" — це вже не зовсім правда після переходу на webhook flow (P1)

### 11.4 ❌ Експорт даних — JSON, нема CSV / GDPR-style (P2)

`HubBackupPanel` — JSON-only. Для GDPR-рекоммендації потрібен:

## 18. Як міряти ефект

- **Activation:** % users з ≥3 записами в перші 7 днів (нині: невідомо, треба PostHog dashboard).
- **Time-to-first-value:** від install до першого запису. Має скоротитись після уніфікації landing-табу і module-settings shortcut'а.
- **Drop-off на FTUX→Day1:** % хто не повернувся на 2-й день. Має покращитись після зменшення signal-noise (R1, R4).
- **Privacy-confidence:** opt-in-rate для App-lock. Якщо <20% — означає, що ще не виставили flow добре.
- **Settings-discovery:** кількість unique sessions, де users відкривали Settings → Notifications. Має зрости після Hub-notifications sub-group.
- **Error-recovery rate:** % сесій з ModuleError, де користувач натискав Retry vs закривав → треба додати телеметрію в ModuleErrorBoundary.

**Підсумково:** UX-фундамент Sergeant'а _зрілий_ — design-system, ARIA, motion-safe, error-boundaries, i18n catalog, empty-state component, hash deep-links у settings. Це 80% від «дуже хорошого продукту». Решту 20% з'їдає **3 кластери проблем**:

1. **Trust** (немає app-lock на приватні фінансово/медичні дані).

3. **Копірайт-баги і tone-of-voice fragmentation** (англ. слова в UA-product, різні стилі loading/empty/validation).

Якщо вирішити ці три речі за 2-3 спринти — буде помітний стрибок activation/retention. Решта recommendations — polish.
