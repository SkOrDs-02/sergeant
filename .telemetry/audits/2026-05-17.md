## Observed Patterns

- **Naming style:** snake_case across all 94 events. Consistent.
- **Naming format:** mixed. Examples: `subject_verb` (`expense_added`, `paywall_viewed`), `action_object` (`auth_after_value`), past-tense status (`sync_succeeded`, `signup_completed`). No single documented convention.
- **Centralization (event names):** strong. Every callsite references `ANALYTICS_EVENTS.<KEY>` — no inline strings found in app code.
- **Centralization (transport):** strong. Single `trackEvent` wrapper per platform: [`analytics.ts`](../../apps/web/src/core/observability/analytics.ts) (web), [`analytics.ts`](../../apps/mobile/src/lib/analytics.ts) (mobile), [`posthogCapture.ts`](../../apps/server/src/lib/posthogCapture.ts) (server).
- **Error handling:** fire-and-forget. Wrapper catches `posthog.capture` throws and continues. No `await` on critical paths.
- **Blocking calls on critical path:** none observed.
- **Shutdown / flush:** not explicit. Browser `beforeunload` flush not configured. Server `posthogCapture` has no shutdown hook.
- **Dual-write:** in addition to PostHog, events go to a localStorage/MMKV ring buffer (200 web / 100 mobile) and a `/recall` AI-memory sink ([`productMemorySync.ts`](../../apps/web/src/core/observability/productMemorySync.ts)).
- **Payload contracts:** documented as JSDoc-style comments inside `analyticsEvents.ts`. Not enforced by types — `trackEvent` signature is `(name, payload?: Record<string, unknown>)`.

## Hygiene Notes

_Factual observations, not recommendations._

1. Mobile has **two identity bridges** in different directories ([`features/analytics/AnalyticsIdentityBridge.tsx`](../../apps/mobile/src/features/analytics/AnalyticsIdentityBridge.tsx), [`observability/IdentityBridge.tsx`](../../apps/mobile/src/observability/IdentityBridge.tsx)) with similar logic — both call `identifyPostHogUser` and `resetPostHog`.
2. No `posthog.group()` calls anywhere. Better Auth account/organization scope is not surfaced as a PostHog group dimension. Server-side capture also omits `groups` in payloads.
3. `Sentry.setUser` is called only on the server. Web + mobile rely on `beforeSend` id-only mapping. PostHog distinct_id and Sentry user.id therefore share the same string but are linked only at the destination, not at the SDK level.
4. Mobile `@sentry/react-native` is declared in [`apps/mobile/package.json`](../../apps/mobile/package.json) but no `sentry*` init file was found under `apps/mobile/`.
5. Server `posthogCapture` fails open when `POSTHOG_PROJECT_API_KEY` is missing — silent skip with no log line.
6. Payload contracts for `SUBSCRIPTION_STARTED` / `SUBSCRIPTION_CANCELED` / `SUBSCRIPTION_RENEWED` in [`analyticsEvents.ts`](../../packages/shared/src/lib/analyticsEvents.ts) are documented as future-state placeholders; current fire-sites in [`apps/server/src/modules/billing/stripe.ts`](../../apps/server/src/modules/billing/stripe.ts) emit shapes derived from Stripe objects.
7. Event naming format is internally mixed (subject_verb vs verb_object vs past-tense status). All snake_case, but no single documented convention.
8. Dual-write to localStorage/MMKV ring buffer and `/recall` endpoint means total instrumented event volume exceeds PostHog event volume. PostHog dashboards reflect a subset.
9. Mobile has no `screen()` equivalent of web's `PageviewTracker` — screen transitions in Expo router are not captured.
10. `EXPERIMENT_EXPOSED` is documented to fire once per render-of-the-arm, but the catalog also has dedicated `_SHOWN` events for some experiments (e.g. `ONBOARDING_GOAL_FIRST_SHOWN`) — both fire for the same event, producing redundant exposure signals for goal-first.
